# 零售端库存管理系统后端需求文档

## 1. 文档定位

### 1.1 目标
- 本文档用于指导零售端库存管理系统后端服务、数据库、接口、安全与部署能力的设计和实施。
- 本文档聚焦服务端架构边界、领域模块、数据模型原则、事务一致性、权限校验、接口规范、文件存储与运维要求。

### 1.2 范围
- 面向移动端 App 提供统一 HTTP/JSON 服务接口。
- 覆盖认证鉴权、仓库权限、商品与库存、单据流转、报表分析、文件上传、审计追溯等能力。
- 为未来 Web 端、多仓汇总、组织扩展与第三方系统对接预留能力。

### 1.3 角色范围说明
- 本期后端权限模型以“管理员”和“普通用户”为主。
- 仓库管理员在一期默认纳入管理员权限体系；后续若需要更细颗粒度授权，可扩展为独立角色。

## 2. 总体架构要求

### 2.1 架构目标
- 后端采用前后端分离架构，为 iOS、Android 及未来 Web 端提供统一的 HTTP/JSON 服务接口。
- 系统部署形态为“客户端 -> 网关 -> 后端应用 -> PostgreSQL / 对象存储”。
- 后端需承担认证鉴权、业务规则校验、库存流水处理、报表计算、文件上传授权与审计追溯等职责。

### 2.4 当前工程技术基线
- 当前后端技术基线采用 Go、Gin、PostgreSQL、JWT、Swagger 与 Docker 等能力作为建设起点。
- 正式建设阶段应尽量沿用当前工程基础设施骨架，以降低切换成本并保持实施连续性。
- 在保持当前技术基线的前提下，接口、模块和数据模型仍需按本文件要求逐步补齐正式业务能力。

### 2.2 分层设计
- 接口层：负责参数接收、校验、响应输出和追踪标识透传。
- 业务层：负责业务规则编排、事务边界控制、幂等策略和跨模块协同。
- 数据访问层：负责数据库读写、查询封装和持久化策略。
- 基础设施层：负责数据库连接、对象存储、鉴权中间件、配置管理、日志监控和通用组件。

### 2.3 架构原则
- 模块化与领域解耦，避免库存、销售、权限、报表等领域逻辑相互耦合。
- 服务端必须作为权限、库存事务、成本价控制和审计落账的最终裁决者。
- 所有关键写操作都必须具备可追溯、可回放、可审计的记录能力。

## 3. 后端模块划分

### 3.1 认证与用户模块
- 负责账号登录、令牌签发与校验、用户信息维护、角色与权限绑定、登录状态管理。
- 支持管理员与普通用户两类主要角色，并预留仓库管理员、审计角色等扩展能力。

### 3.2 仓库与权限模块
- 负责仓库主数据维护、用户与仓库绑定关系、默认仓库、当前仓库上下文切换。
- 普通用户只允许绑定一个仓库。
- 管理员可绑定多个仓库，并在授权范围内切换。
- 所有接口都必须附加仓库范围校验。

### 3.3 商品与库存模块
- 负责商品档案、条码/二维码绑定、标准库存与非标库存管理、库存预警和库存状态展示。
- 所有库存数据必须带有明确仓库归属，并可追踪库存变化来源。

### 3.4 单据与业务流水模块
- 应支持入库单、销售单、退货单、调拨单、盘点单、非标转换单等业务单据。
- 每类业务都应形成单据主表、明细表与库存流水记录。
- 单据流转必须支持状态追踪、操作者追踪、时间追踪和来源追踪。

### 3.5 报表与分析模块
- 提供销售统计、库存分析、排行榜、趋势图、库存周转率、滞销预警等能力。
- 报表数据必须受角色与仓库权限约束，避免出现跨仓越权统计。

### 3.6 文件与附件模块
- 负责商品图片、附件、导入模板、导出结果等文件的上传授权、访问控制、元数据存储与生命周期管理。

### 3.7 审计与日志模块
- 负责关键操作审计、接口调用日志、异常日志、任务执行日志和安全事件记录。
- 支持按用户、仓库、商品、单据号和时间范围查询审计记录。

## 4. 业务规则的服务端落地要求

### 4.1 角色与仓库规则
- 普通用户只能访问并操作绑定仓库内的数据。
- 管理员只能访问其授权仓库集合中的数据。
- 所有库存、销售、退货、盘点、调拨、报表、导出接口都必须执行角色和仓库双重校验。

### 4.2 标准库存与非标库存规则
- 非标库存只能由管理员创建、调整和转换。
- 非标库存不得参与销售出库。
- 非标库存不得跨仓调拨。
- 非标库存支持部分转换为标准库存，未转换部分需继续保留为非标数量。
- 转换后参与销售和调拨的只能是标准库存。

### 4.3 单据规则
- 销售出库仅允许扣减标准库存。
- 退货入库必须关联原销售单并校验可退数量、状态与仓库归属。
- 调拨仅允许标准库存参与，且仅管理员可执行。
- 盘点单需体现“盘点录入 -> 差异确认 -> 盘点结转”的状态流转。
- 非标转换单需体现“选择非标 -> 填写数量 -> 关联目标标准商品 -> 完成转换 -> 记录转换结果”的业务闭环。

## 5. 数据库设计要求

### 5.1 数据库目标
- 优先保证事务一致性、查询性能、扩展能力与运维可管理性。
- 所有核心业务变化都必须可记录、可查询、可追溯。

### 5.2 核心实体
- 用户表
- 角色表与权限表
- 仓库表
- 用户仓库绑定表
- 商品表
- 库存表
- 非标库存表
- 单据主表与单据明细表
- 库存流水表
- 图片资源表
- 审计日志表

### 5.3 建模原则
- 核心业务表需包含主键、创建时间、更新时间、创建人、更新人等基础字段。
- 单据和库存表必须体现仓库维度。
- 商品编码、条码、临时标签号、单据号等关键字段需建立唯一约束。
- 高频查询字段需建立索引，如仓库 ID、商品 ID、条码、单据号、创建时间、操作人和业务状态。
- 库存流水、审计日志、报表明细等高增长表需预留时间分区、归档或冷热分层策略。

### 5.4 数据一致性要求
- 入库、销售出库、退货入库、盘点结转、非标转标准、跨仓调拨等关键写操作必须在事务内处理。
- 必须保证单据写入、库存扣减或回补、库存流水和审计记录一致提交或一致回滚。
- 对普通用户不可见的数据，如成本价和非授权仓库数据，数据库查询与应用层返回都应执行双重限制。

## 6. 数据域与数据口径

### 6.1 用户与权限数据
- 保存用户基础信息、登录标识、角色信息、用户状态、所属组织或门店信息、最后登录时间。
- 保存角色、权限点、角色权限映射、用户仓库绑定关系和默认仓库。

### 6.2 仓库与主数据
- 保存仓库编码、名称、状态、地址、负责人和联系方式。
- 保存商品名称、分类、规格、单位、系统商品 ID、条码/二维码、建议零售价、成本价控制信息和图片关联信息。
- 保存单位、商品分类、库存状态、单据状态、操作类型、预警等级等字典数据。

### 6.3 库存与单据数据
- 标准库存数据需记录仓库归属、可用库存、锁定库存、库存状态、预警阈值和更新时间。
- 非标库存需记录临时标签号、仓库、临时描述、单位、数量、来源方式、来源单据、创建人、创建时间、当前状态等信息。
- 单据和库存流水需记录每次库存变化的前后数量、变化方向、业务类型、来源单据、操作者和发生时间。

### 6.4 数据约束
- 所有库存类、单据类、流水类数据必须带有仓库归属。
- 销售单与退货单之间必须有关联关系。
- 非标库存与转换记录之间需支持一对多或多次转换关系。
- 报表、图表、列表与详情页使用的状态字段和统计口径必须一致。

## 7. 接口与通信规范

### 7.1 接口风格
- 客户端与服务端统一采用 HTTP/JSON 通信。
- 接口风格以 REST 为主。
- API 应采用版本化路径，例如 `/api/v1`。

### 7.2 查询接口要求
- 支持分页、关键字搜索、条件筛选、排序和按仓库过滤等通用能力。
- 报表查询接口需明确统计口径、时间范围、权限边界和聚合维度。

### 7.3 写入接口要求
- 明确区分“创建单据”“提交审核”“执行转换”“确认盘点”等业务动作。
- 对批量导入、批量出库、批量盘点等长耗时场景，需提供异步任务、进度反馈和结果查询机制。
- 高风险写接口必须具备参数校验、权限校验、状态校验、幂等控制和审计记录。

### 7.4 统一响应与错误规范
- 响应结构至少包括业务状态码、提示信息、业务数据和追踪标识。
- 错误码需覆盖认证失败、权限不足、参数校验失败、库存不足、状态不允许、资源不存在、重复提交和系统异常等场景。
- 错误提示需便于客户端直接展示，同时避免暴露内部路径、数据库结构或敏感实现细节。

### 7.5 接口文档要求
- 需提供 Swagger / OpenAPI 文档，并与实际发布版本保持同步。
- 开发环境需支持基于接口文档的调试与联调。

## 8. 幂等、事务与异步任务

### 8.1 幂等要求
- 销售出库、退货入库、盘点结转、非标转标准、调拨、批量导入等关键操作必须支持幂等控制。
- 同一业务动作因弱网重试、用户重复点击或请求重放不得重复入账。
- 需要保存请求级幂等标识或业务级去重依据。

### 8.2 事务要求
- 事务边界应覆盖单据主表、明细表、库存记录、库存流水和审计记录。
- 异常中断时应保证完整回滚，避免半成功状态。

### 8.3 异步任务要求
- 长耗时操作需返回任务 ID。
- 服务端需提供任务状态查询、进度反馈和结果回看能力。
- 对失败任务需提供可恢复语义和错误原因。

## 9. 安全与权限技术要求

### 9.1 认证机制
- 采用令牌鉴权机制。
- 登录成功后签发访问令牌，令牌仅包含最小必要身份信息。
- 支持令牌过期控制与续签机制。

### 9.2 授权机制
- 采用基于角色的权限控制模型。
- 所有库存、销售、退货、调拨、盘点、非标转换、报表查询和导出接口都必须在服务端执行权限校验。
- 不得仅依赖客户端隐藏按钮或隐藏字段实现权限控制。

### 9.3 数据安全
- 生产环境必须使用 HTTPS/TLS。
- 密码、密钥、数据库凭证等敏感配置需通过环境变量或安全配置中心管理，不得硬编码。
- 成本价、密钥、审计信息等敏感字段需按最小权限原则控制读取范围。

### 9.4 审计与防护
- 关键写入操作应记录审计日志，包括操作者、仓库、业务类型、目标对象、结果、时间和来源终端。
- 需具备参数校验、越权拦截、防重复提交、基础限流、防重放和异常告警能力。
- 对导入、上传、批量操作等高风险接口，应增加额外校验、大小限制和追踪记录。

### 9.5 备份与恢复
- 关键业务数据需建立备份、恢复、保留与归档策略。
- 必须定期验证恢复能力，确保历史数据可查询、可追溯。

## 10. 文件存储与附件管理

### 10.1 存储原则
- 商品图片、附件、导入模板和导出结果等文件应保存在对象存储中。
- 数据库只保存文件 URL、对象键、业务类型、上传人、上传时间、文件大小、文件哈希和 MIME 类型等元数据。

### 10.2 上传与访问控制
- 上传接口需校验文件类型、大小、命名规则和业务归属。
- 文件访问需区分公开资源与受控资源。
- 受控资源可通过签名 URL、短期访问凭证或服务端代理下载方式控制。
- 删除业务数据或图片时，应定义元数据删除与对象回收策略。

### 10.3 扩展要求
- 支持后续扩展缩略图、图片压缩、水印、批量导出与 CDN 加速等能力。

## 11. 部署与运行环境要求

### 11.1 开发环境
- 需支持本地快速启动数据库等基础依赖。
- 需提供接口文档访问入口。
- 配置管理以环境变量为主，保证开发、测试、生产环境分离。

### 11.2 测试与生产环境
- 需支持容器化打包与部署。
- 网关作为统一入口，承担反向代理、HTTPS 终止与负载均衡职责。
- 数据库需配置持久化存储、健康检查、备份策略与恢复演练机制。

### 11.3 配置与初始化
- 数据库初始化脚本、迁移脚本和部署编排需与工程目录保持一致。
- 需建立明确的数据库迁移策略，支持新环境初始化、升级、回滚和补丁发布。

## 12. 性能、可用性与演进要求

### 12.1 性能目标
- 常用查询响应目标不超过 1 秒。
- 列表与检索场景首屏目标不超过 2 秒。
- 单据类写入目标在 3 秒内返回明确结果。
- 大批量操作总体目标在 30 秒内完成，并具备进度反馈。
- 系统需支持不少于 30 并发用户的门店或仓库场景。

### 12.2 可用性目标
- 高峰期优先保证写入类操作的可靠性与一致性。
- 支持静态或低变更数据的缓存，并在变更后及时失效。
- 对网络抖动接口设置合理超时和重试策略。

### 12.3 演进要求
- 支持新增库存状态、新单据类型、新报表维度和第三方系统对接字段。
- 枚举、状态码、操作类型、业务标签应优先采用字典化和配置化管理。
- 对高并发、高数据量场景逐步补充缓存、异步任务、监控告警、统计汇总表与历史归档能力。

## 13. 项目交付与验收

### 13.1 后端阶段交付物
- 数据库设计文档
- 接口文档
- 迁移脚本
- 核心模块实现
- 基础部署说明
- 日志与监控接入方案

### 13.2 后端验收重点
- 认证鉴权、角色权限、仓库隔离、库存事务和单据流转规则准确落地。
- 关键接口事务控制、幂等校验、审计记录和错误码规范完整可验证。
- 报表、导出、列表、详情等查询口径一致。
- 上线前完成灰度、监控、告警与回滚预案验证。

## 14. 风险与应对

### 14.1 数据一致性风险
- 若事务控制、幂等校验或异常补偿不足，会导致单据、库存、流水与审计记录不一致。
- 应对：对关键写操作统一实施事务控制、幂等机制与异常回滚，并覆盖重复提交、弱网重试和异常中断测试。

### 14.2 权限与越权风险
- 若接口层、导出层、查询层权限口径不一致，可能出现越权查看和跨仓操作。
- 应对：在服务端、导出逻辑和查询封装中统一执行角色与仓库范围校验，并保留审计日志。

### 14.3 敏感数据泄露风险
- 成本价、令牌、审计信息和内部配置若控制不严，会引发合规与经营风险。
- 应对：按最小权限原则控制字段、缓存、导出和日志输出范围，并建立专项检查机制。

### 14.4 交付协同风险
- 若接口文档、字段定义、状态口径和验收标准不同步，容易造成联调反复。
- 应对：建立固定联调评审机制，保持需求文档、接口文档、数据库设计与测试用例联动更新。
