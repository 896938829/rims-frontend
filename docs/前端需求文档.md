# 零售端库存管理系统前端需求文档

## 1. 文档定位

### 1.1 目标
- 本文档用于指导零售端库存管理系统移动端 App 的前端设计与实施。
- 本文档聚焦页面结构、交互规则、权限显隐、终端能力、弱网策略、埋点规范以及与后端接口协作约束。

### 1.2 适用范围
- 适用于 iOS、Android 以及鸿蒙设备 Android 兼容层。
- 本阶段不包含独立 Web 端交付，但页面结构、设计语言与接口契约需为后续 Web 扩展预留基础。

### 1.4 角色范围说明
- 本期前端权限矩阵以“管理员”和“普通用户”为主。
- 仓库管理员在一期默认按管理员能力模型展示和管控，后续如拆分独立角色，再补充专属权限矩阵。

### 1.3 前端职责边界
- 负责界面组织、导航路由、交互反馈、状态展示、本地缓存和终端能力调用。
- 负责表单前置校验、错误映射、弱网提示、草稿恢复、埋点与终端监控。
- 权限校验、库存事务、成本价控制、审计记录、幂等裁决最终以服务端返回结果为准。

## 2. 前端总体目标

### 2.1 建设目标
- 提供覆盖登录鉴权、仓库切换、库存管理、销售出库、退货入库、盘点、非标转标准、报表分析、扫码与附件上传等核心流程的移动端操作界面。
- 保证关键流程少步骤可达、状态可识别、反馈清晰、弱网可恢复。
- 保证不同角色在菜单、字段、操作和导出口径上的展示一致。

### 2.2 设计原则
- 简洁、专业、现代、扁平化。
- 高信息密度但低认知负担。
- 优先适配触屏高频作业场景。
- 所有关键业务状态必须文案、标签、图标协同表达，不得仅依赖颜色。

## 3. 客户端技术路线与架构

### 3.1 技术路线
- 移动端以 Godot 4.x 稳定版本作为主要 UI 开发工具，承载页面布局、主题系统、动画、图表、资源管理与跨平台导出能力。
- UI 与业务编排优先采用 GDScript 实现，以降低界面迭代成本。
- 性能敏感或需要调用系统能力的部分，可通过 Godot Android Plugin、iOS 原生桥接或 GDExtension 完成。
- 客户端通过版本化 HTTP/JSON 接口访问认证、商品、库存、单据、报表和文件上传能力。

### 3.4 当前工程与导出基线
- 工程组织建议至少区分 `scenes`、`ui_components`、`themes`、`scripts`、`services`、`models`、`assets`、`plugins`、`locales`、`tests` 等目录。
- Android 端通过官方导出链路生成调试包和正式包，并接入必要原生插件支撑扫码、相机、相册、文件选择和安全存储。
- iOS 端通过导出工程接入签名、权限声明、图标和启动配置。
- 鸿蒙设备兼容层优先复用 Android 包进行测试和发布验证，并单独记录兼容差异。

### 3.2 分层结构
- 表现层：页面场景、控件组件、主题换肤、弹窗、图表、提示与动效。
- 应用层：导航路由、页面状态编排、表单校验、请求发起、错误映射、上传流程、重试队列。
- 领域层：角色菜单裁剪、库存状态标签映射、单据状态文案、仓库上下文管理、图表指标格式化。
- 基础设施层：网络请求封装、令牌持久化、安全存储、日志埋点、扫码桥接、相机/相册/文件访问、声音与触感反馈。

### 3.3 模块划分
- 认证与会话模块
- 首页与导航模块
- 库存与商品模块
- 单据与业务操作模块
- 报表与图表模块
- 终端能力模块

## 4. 终端适配与布局策略

### 4.1 设备适配
- 优先适配主流 Android 手机、iPhone、iPad 及常见安卓平板分辨率。
- 鸿蒙设备以 Android 兼容层运行 Android 包的方式纳入本期支持范围。
- 页面需适配异形屏、安全区、系统状态栏和手势返回区域，避免关键操作区域被遮挡。

### 4.2 横竖屏策略
- 手机默认竖屏，保障登录、扫码、销售、退货、盘点录入等高频流程单手可达。
- 图表、长表格与盘点对账场景允许横屏增强显示。
- 平板需支持横竖屏，横屏优先采用“主列表 + 详情”或“左侧导航 + 右侧内容区”布局。
- 横竖屏切换时尽量保留仓库上下文、查询条件、草稿数据和滚动位置。

### 4.3 响应式布局要求
- 小屏手机采用单栏流式布局，底部一级导航不超过 5 个。
- 大屏平板允许同屏展示筛选、列表、详情、统计摘要与辅助说明。
- 图表页面在平板横屏下优先展示更多筛选和多图联动能力。
- 长表单页面优先使用分组卡片、分步流程或底部弹层，而不是一次性铺满全屏。

## 5. 视觉与组件规范

### 5.1 视觉风格
- 强调简洁、专业、现代、扁平化。
- 通过字号、字重、留白、分组边界与色彩语义体现信息层级。
- 图标风格统一，状态图标需与业务语义保持一致。

### 5.2 双主题要求
- 支持浅色主题、深色主题和跟随系统三种模式。
- 两套主题下的标准库存、非标库存、已转换标准、失败状态、禁用状态等语义标签含义必须完全一致。

### 5.3 组件规范
- 统一按钮、输入框、搜索框、筛选器、分页加载、卡片、标签、对话框、底部操作栏、空状态、错误页和 Toast 的样式与状态机。
- 触控控件必须具备按下、禁用、加载中和完成状态。
- 关键按钮在请求处理中必须加载锁定，避免重复提交。

## 6. 信息架构与页面设计

### 6.1 一级导航
- 首页
- 库存
- 业务单据
- 报表
- 我的

### 6.2 角色化首页
- 管理员首页突出仓库切换、库存预警、待处理单据、报表摘要与快捷业务入口。
- 普通用户首页突出当前仓库、扫码销售、退货入库、库存查询和最近操作记录。
- 所有涉及库存、商品、单据、报表的页面都必须明确展示当前仓库上下文。

### 6.3 核心页面清单
- 登录与身份认证页
- 仓库选择与切换页
- 库存总览页
- 商品详情与商品维护页
- 销售出库页
- 退货入库页
- 入库页
- 调拨页
- 盘点页
- 非标转标准页
- 报表与分析页
- 设置与“我的”页

### 6.4 状态展示原则
- 标准库存、非标库存、已转换标准等状态需通过标签、图标与文案双重表达。
- 成本价、跨仓汇总、敏感审计信息等受控字段需按角色显隐，并提供合理的不可见说明。
- 对库存不足、状态不允许、权限不足、网络失败、重复提交等场景应使用统一术语和统一错误映射策略。

## 7. 关键交互规则

### 7.1 触屏交互
- 高频触控目标应具备适合移动端的点击热区尺寸。
- 关键入口尽量靠近拇指易达区域。
- 列表支持下拉刷新、分页加载、惯性滚动、吸顶筛选栏和空态提示。

### 7.2 表单与单据交互
- 表单需支持必填、格式、范围、数量上限、价格异常、仓库权限、库存状态等前置校验。
- 长流程页面按“基础信息 -> 明细 -> 结果确认 -> 提交反馈”组织。
- 销售出库、退货入库、盘点结转、非标转标准、调拨提交等关键动作必须在提交前展示受影响的仓库、商品、数量与状态，并提供二次确认。
- 提交按钮必须与请求幂等标识绑定，避免重复入账。

### 7.3 扫码与图片识别
- 首页、库存查询、销售出库、盘点录入等高频场景要保证扫码入口就近可达。
- 支持一维码、二维码识别。
- 支持相册图片识别作为备用路径。
- 扫码失败、匹配多个商品或识别到不可售非标库存时，必须给出明确后续指引。
- 相机、相册、文件访问等权限需按需申请，并在被拒绝后提供重新授权说明。

### 7.4 错误处理与可恢复性
- 网络失败、超时、权限不足、库存不足、状态不允许、数据过期、资源不存在、重复提交等错误需直接展示用户可理解文案。
- 对批量导入、批量盘点、长耗时报表等场景，需提供进度展示、状态轮询、失败重试和结果回看能力。
- 弱网或离线状态下允许浏览本地缓存的商品、库存与报表摘要。
- 所有提交类操作必须明确当前网络状态及是否允许发起。

## 8. 关键业务流程

### 8.1 登录流程
- 输入手机号验证码或账号密码。
- 本地校验输入格式。
- 调用登录接口。
- 获取用户身份、角色、仓库权限和首页初始化所需数据。
- 登录成功后进入对应角色首页。
- 鉴权失效时需保留必要草稿和页面意图，重新登录后尽量恢复原操作上下文。

### 8.2 用户类型确认与首页初始化
- 当前若账号仅绑定单一角色，则登录成功后直接进入对应首页。
- 若未来支持一个账号多身份，则需提供用户类型确认页。
- 首页初始化必须完成角色菜单裁剪、仓库上下文设置、快捷入口渲染、消息与预警摘要加载。

### 8.3 仓库选择与切换
- 普通用户固定绑定仓库，仅展示仓库名称，不展示切换入口。
- 管理员可查看授权仓库列表、搜索最近使用仓库并切换当前仓库。
- 切换成功后刷新库存、单据、报表和快捷入口上下文。
- 切换失败时保持原上下文不变并提示原因。

### 8.4 销售出库流程
- 从首页快捷入口、库存页或扫码入口进入。
- 选择或继承当前仓库上下文。
- 通过扫码或手动选商品录入明细。
- 仅展示可销售的标准库存。
- 输入数量和实际销售价后，展示建议零售价对比与库存校验结果。
- 二次确认后提交销售单并返回成功回执。
- 失败时要明确区分库存不足、非标不可售、价格校验失败、重复提交和网络中断等原因。

### 8.5 退货入库流程
- 从销售记录详情、首页快捷入口或单据中心进入。
- 查询或扫码关联原销售单。
- 选择可退商品和数量。
- 校验可退数量、状态与仓库归属。
- 填写退货原因并提交。
- 返回库存回写结果与失败原因。

### 8.6 报表查看与导出流程
- 进入报表页后选择时间范围、仓库范围和图表类型。
- 拉取统计结果并支持图表与表格切换核对。
- 执行导出或分享时需再次遵循权限口径。
- 涉及成本价、多仓汇总等敏感内容时，前端仅做提示，最终是否允许导出以后端校验结果为准。

### 8.7 管理员专属流程
- 入库流程：支持扫码、手动、批量导入三种路径。
- 调拨流程：仅允许标准库存参与，需明确调出仓、调入仓、商品和数量。
- 盘点流程：体现“盘点录入 -> 差异确认 -> 盘点结转”的三步闭环。
- 非标转标准流程：体现“选择非标记录 -> 输入转换数量 -> 关联标准商品 -> 二次确认 -> 完成转换”的路径。

## 9. 权限矩阵与字段显隐

### 9.1 前端权限原则
- 前端权限控制用于页面入口显隐、按钮可用性、字段可见性和提示信息表达。
- 无权限能力优先采用隐藏入口或置灰并说明原因的方式呈现。
- 任何关键受控能力都不得仅依赖前端限制，必须与后端权限校验协同。

### 9.2 角色权限矩阵
| 功能域 | 页面或入口 | 管理员 | 普通用户 | 说明 |
|------|------------|--------|----------|------|
| 登录鉴权 | 登录、退出、会话续签 | 可用 | 可用 | 所有用户通用 |
| 仓库上下文 | 仓库切换、默认仓库设置 | 可用 | 不可切换 | 普通用户固定绑定仓库 |
| 首页快捷入口 | 入库、调拨、盘点、非标转标准 | 可见 | 隐藏 | 管理员专属 |
| 首页快捷入口 | 销售出库、退货入库、库存查询 | 可见 | 可见 | 普通用户核心能力 |
| 商品信息 | 基础信息、建议零售价、图片 | 可见 | 可见 | 按仓库范围控制 |
| 商品信息 | 成本价、平均售价 | 可见 | 隐藏 | 受控字段 |
| 库存管理 | 标准库存查看 | 可见 | 可见 | 受仓库权限限制 |
| 库存管理 | 非标库存查看与处理 | 可见 | 仅摘要或隐藏 | 普通用户不得调整非标库存 |
| 单据操作 | 普通入库 | 可用 | 不可用 | 指新增库存、批量入库、盘点结转入库等 |
| 单据操作 | 销售出库 | 可用 | 可用 | 仅允许标准库存参与销售 |
| 单据操作 | 退货入库 | 可用 | 可用 | 必须关联销售记录 |
| 单据操作 | 调拨、盘点、非标转标准 | 可用 | 不可用 | 管理员专属 |
| 报表分析 | 单仓报表查看 | 可用 | 可用 | 普通用户限绑定仓库 |
| 报表分析 | 多仓汇总、库存成本分析 | 可用 | 隐藏 | 仅管理员可见 |
| 导出分享 | 普通报表导出 | 可用 | 可用 | 导出内容受权限口径限制 |
| 导出分享 | 含成本价或多仓敏感数据导出 | 可用 | 不可用 | 服务端二次校验 |

### 9.3 字段级显隐规则
- 成本价：管理员显示，普通用户隐藏。
- 平均售价：管理员显示，普通用户隐藏或不展示。
- 多仓汇总数据：管理员显示，普通用户隐藏。
- 非标处理按钮：管理员显示，普通用户隐藏。
- 仓库切换控件：管理员显示，普通用户隐藏并展示固定仓库名称。
- 审计摘要：按授权显示，普通用户隐藏。

## 10. 与后端协作约束

### 10.1 接口契约原则
- 客户端通过版本化 HTTP/JSON 接口与服务端通信。
- 所有接口需遵循统一响应结构，至少包含业务状态码、提示信息、业务数据与追踪标识。
- 客户端错误提示需基于统一错误码映射，而不是直接拼接后端原始异常。

### 10.2 前端必须对齐的业务口径
- 普通用户只能操作绑定仓库。
- 销售出库仅允许标准库存参与。
- 退货入库必须关联原销售记录。
- 调拨仅允许标准库存且仅管理员可执行。
- 非标库存只能由管理员创建、调整和转换。
- 报表和导出必须与页面展示遵循同一权限口径。

### 10.3 需要补齐的联调交付物
- 接口文档或 OpenAPI 文档。
- 统一错误码表与前端展示文案映射。
- 关键状态枚举表，如库存状态、单据状态、操作类型、预警等级。
- 长耗时任务接口约定，包括任务 ID、状态查询、进度反馈和失败恢复语义。

## 11. 隐私、缓存与埋点

### 11.1 用户信息与本地存储边界
- 本地仅保存支撑登录鉴权、权限展示、业务流转和连续使用体验所必须的信息。
- 密码、验证码、访问令牌、刷新令牌、成本价、敏感审计信息不得明文写入日志、埋点或可被普通用户导出的本地文件。
- 退出登录、切换账号或服务端鉴权失效时，需清除该账号对应的本地敏感缓存与草稿关联信息。

### 11.2 缓存策略
- 商品、库存、报表摘要等缓存需带有效期与更新时间。
- 仓库切换、账号切换、权限刷新后必须同步清理或失效相关缓存。
- 关键状态字段需尽量与服务端真实状态保持一致，避免缓存延迟引起跨仓误判或状态误判。

### 11.3 埋点规范
- 埋点用于分析登录转化、功能使用、关键链路成功率、弱网问题、扫码识别效果和报表导出使用情况。
- 公共字段建议包含事件时间、用户 ID、角色类型、当前仓库 ID、终端平台、系统版本、应用版本、页面标识、网络类型、主题模式等。
- 不得采集密码、验证码、完整手机号、完整条码内容、成本价明细、敏感审计详情等敏感内容。

## 12. 发布、运维与质量要求

### 12.1 环境与版本
- 客户端需区分开发、测试、预发布和生产环境。
- 版本需与接口版本、数据库变更窗口和功能开关保持可追踪关系。

### 12.2 运维能力
- 接入崩溃监控、性能监控、行为埋点与关键业务日志上报能力。
- 对高风险能力支持远程配置或特性开关。
- 客户端日志与埋点不得泄露受控信息。

### 12.3 升级与回滚
- 支持强制升级与建议升级。
- 出现严重兼容性问题时，应支持通过应用市场回滚、关闭特性开关或切换服务端兼容策略止损。

## 13. 前端验收重点

### 13.1 流程验收
- 登录、首页初始化、仓库切换、销售出库、退货入库、报表查看与导出、入库、调拨、盘点、非标转标准等关键流程必须可演示、可回归。

### 13.2 权限验收
- 页面入口、按钮、字段、导出和提示信息必须符合角色与仓库范围口径。

### 13.3 终端验收
- Android、iOS、鸿蒙 Android 兼容层关键业务流程需完成真机验证。
- 弱网、离线浏览、重复点击、扫码失败、相册权限拒绝等场景需完成专项验证。

### 13.4 联调验收
- 前端展示状态、报表口径、错误提示与后端接口返回保持一致。
- 长耗时任务、批量操作、导出和上传流程具备可恢复性与明确反馈。
